## 一、总体架构分层

OpenClaw 的 `src/` 采用 **分层 + 领域驱动** 的组织方式，从上到下依次为：

```mermaid
flowchart TB
    subgraph EntryLayer["入口层"]
        Entry["entry.ts<br/>CLI 入口"]
        Index["index.ts<br/>库入口 / API 导出"]
    end

    subgraph CLILayer["CLI 层"]
        CLI["cli/<br/>程序构建 + 子命令"]
        Commands["commands/<br/>命令实现"]
    end

    subgraph GatewayLayer["Gateway 层"]
        Gateway["gateway/<br/>WebSocket 服务 + HTTP API"]
        ACP["acp/<br/>Agent Control Plane"]
    end

    subgraph AgentLayer["Agent 运行时层"]
        Agents["agents/<br/>核心运行时（最大模块）"]
        CtxEngine["context-engine/<br/>上下文引擎"]
        Memory["memory/<br/>记忆系统"]
    end

    subgraph ChannelLayer["消息渠道层"]
        Telegram["telegram/"]
        Discord["discord/"]
        Slack["slack/"]
        Signal["signal/"]
        Web["web/<br/>WhatsApp"]
        IMessage["imessage/"]
        Line["line/"]
    end

    subgraph AbstractLayer["渠道抽象层"]
        Channels["channels/<br/>渠道接口"]
        Routing["routing/<br/>消息路由"]
        AutoReply["auto-reply/<br/>自动回复管线"]
    end

    subgraph PluginLayer["插件层"]
        Plugins["plugins/<br/>插件运行时"]
        PluginSDK["plugin-sdk/<br/>开发 SDK"]
        Hooks["hooks/<br/>Hook 加载"]
    end

    subgraph InfraLayer["基础设施层"]
        Infra["infra/<br/>环境 + 执行"]
        Config["config/<br/>配置管理"]
        Process["process/<br/>进程管理"]
        Logging["logging/<br/>日志"]
        Terminal["terminal/<br/>终端输出"]
        Secrets["secrets/<br/>密钥管理"]
        Security["security/<br/>安全审计"]
    end

    subgraph FeatureLayer["功能模块层"]
        Browser["browser/<br/>浏览器自动化"]
        Canvas["canvas-host/<br/>Canvas UI"]
        Cron["cron/<br/>定时任务"]
        Daemon["daemon/<br/>系统服务"]
        Media["media/<br/>媒体处理"]
        TUI["tui/<br/>终端 UI"]
        Wizard["wizard/<br/>引导向导"]
    end

    EntryLayer --> CLILayer
    CLILayer --> GatewayLayer
    GatewayLayer --> AgentLayer
    AgentLayer --> AbstractLayer
    AbstractLayer --> ChannelLayer
    PluginLayer --> AbstractLayer
    InfraLayer --> AgentLayer
    FeatureLayer --> AgentLayer
```

## 二、入口文件

### `entry.ts` — CLI 启动入口

```typescript
// 职责链：
// 1. Node.js 版本检查（MIN_NODE_MAJOR = 22）
// 2. Windows argv 规范化
// 3. --profile / --dev 环境变量注入
// 4. ExperimentalWarning 静默过滤
// 5. respawn 策略（必要时重新启动子进程）
// 6. 快速路径：--version / --help 直接响应
// 7. 调用 runCli() 进入主流程

```

供 `extensions/` 目录下的插件包导入使用的统一 API 接口。

| 根级 `*.ts` | System Prompt、工具管线、压缩、策略等核心逻辑 |

## 十四、Swabble 语音唤醒守护进程

`Swabble/` 是仓库根目录下的一个 **独立 Swift 包**（非 `src/` 的一部分），但与 OpenClaw 紧密集成：

```
Swabble/
├── Sources/
│   ├── swabble/              # CLI 入口（Commander 框架）
│   │   ├── main.swift
│   │   └── Commands/         # serve / transcribe / doctor 等
│   ├── SwabbleCore/          # 核心库
│   │   ├── Config/           #   配置管理
│   │   ├── Hooks/            #   Hook 执行器
│   │   └── Support/          #   转录存储
│   └── SwabbleKit/           # 跨平台唤醒词门控工具
├── Tests/
├── Package.swift             # Swift Package Manager
└── docs/spec.md              # 设计规范
```

**作用**：macOS 语音唤醒守护进程，使用 Apple Speech.framework 本地模型持续监听麦克风，检测到唤醒词 "clawd" 后将转录文本通过 Shell Hook 传递给 OpenClaw Agent。

**工作流**：
```
麦克风 → AVAudioEngine → Speech.framework → 唤醒词检测 → Hook → openclaw agent --message "${text}"
```
